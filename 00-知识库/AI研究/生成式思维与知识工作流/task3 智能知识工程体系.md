第三维度：交互协议的进化——S.C.O.R.E. 模型
（对应文档 4.3 章节）
Task02 我们学了 CO-STAR，这里提出了一个更适合系统化工程的模型：S.C.O.R.E.。
S (Setting 设定)： 也是角色设定，但更强调系统视角。
C (Context 背景)： 防止断链推理，喂给它所有已知信息。
O (Objective 目标)： 动词驱动，比如“提取-分析-生成”。
R (Requirements 要求)： 质量护栏，方便自动化。
E (Evaluation 评估)： 这是新重点！ 让 AI 在输出前自检，或者输出后由另一个 AI 评分。
第四维度：终极形态——代理式工作流 (Agentic Workflows)
（对应文档 4.4 章节）
这是本章最激动人心的部分。我们不再是“还是原来的配方（聊天）”，而是构建自主智能体。
什么叫 Agent（智能体）？
传统工作流： 你点一下，动一下。
代理工作流： 你给一个目标（比如“写一份关于 RAG 的行业调研”），Agent 自己去规划（Planning）、自己用工具（Tool Use，比如去谷歌搜索、去读你的笔记）、自己反思修正（Reflection），最后把结果给你。
三种核心模式：
Planner（规划者）： 把大任务拆成小任务。
Tool User（工具人）： 调用搜索、计算器、代码解释器。
Critic（反思者）： 自己骂自己，然后改得更好。
第五维度：运维体系——Knowledge Ops
（对应文档 4.5 章节）
当你有了很多 Agent 和大量笔记，怎么管理？像管理软件项目一样管理知识。
IssueOps： 用 GitHub Issues 来提需求。比如提一个 Issue：“整理本周大模型论文”，系统自动触发 Agent 去干活。
CI/CD（持续集成）： 每次你更新笔记，后台自动运行脚本检查链接有没有断、格式对不对，甚至自动生成新的知识图谱。
总结：Task 03 的核心任务
如果说 Task 02 是让你学会怎么说话（Prompt Engineering），Task 03 就是让你学会怎么盖房子（System Engineering）。
你需要做的心态转变：
从 “写笔记” 转向 “维护知识库”。
从 “问 AI” 转向 “编排 AI”。
接下来的实操建议（预告）：
根据这个理论，你的作业（Task 03 Assignment）很可能是要求你：
配置环境： 在 Obsidian 里配置好 Git 和 YAML 模板。
重构笔记： 把几篇长文章拆解成带元数据的原子笔记。
设计流程： 设计一个简单的 S.C.O.R.E. 提示词，让 AI 基于你的原子笔记回答问题（这是 RAG 的手动模拟）。
